# AeroGuard-MAS - Multi-Agent Airspace Conflict Manager

**AeroGuard-MAS** is a university project for **Intelligent Systems Engineering**.
It implements a simulated multi-agent system for detecting, reasoning about, resolving, explaining, and visualizing airspace conflicts between aircraft.

The project combines:

* **Kotlin** for the core simulation and orchestration.
* **Jason / AgentSpeak(L)** for BDI agent models.
* **tuProlog** for symbolic safety reasoning.
* **STRIPS-style planning** for resolution plans.
* **JSON** for input scenarios.
* **JSONL** for structured simulation events.
* **Python + Streamlit** for the replay GUI.
* **Gradle** for build automation.
* **JUnit 5** for automated testing.
* **GitHub Actions** for CI/CD.
* **MkDocs + Mermaid** for project documentation.
* **Dokka** for Kotlin API documentation.

> AeroGuard-MAS is a didactic simulation. It is **not** a certified aviation or air traffic control system.

## Generating JSONL Events

The Kotlin core emits structured simulation events in JSONL format.

Example event:

```json
{"tick":0,"type":"aircraft_state","aircraft":"AZA123","x":0.0,"y":0.0,"altitude":30000,"speed":1.0,"status":"normal","priority":"NORMAL"}
```

Generate events for the simple conflict scenario:

```bash
./gradlew run --args="--scenario scenarios/simple_conflict.json --events build/aeroguard/events/simple_conflict_events.jsonl --explain"
```

Generate events for weather replanning:

```bash
./gradlew run --args="--scenario scenarios/weather_replanning.json --events build/aeroguard/events/weather_replanning_events.jsonl --explain"
```

Generate events for secondary conflict prevention:

```bash
./gradlew run --args="--scenario scenarios/secondary_conflict.json --events build/aeroguard/events/secondary_conflict_events.jsonl --explain"
```

## Scenarios

The project includes five main demonstration scenarios.

### `simple_conflict.json`

Two aircraft converge toward a loss of separation.
The system should detect a future conflict and generate a simple resolution maneuver.

### `emergency_priority.json`

One aircraft has higher priority due to emergency-related conditions.
The system should prefer maneuvering the lower-priority aircraft.

### `secondary_conflict.json`

A naive maneuver would solve a primary conflict but create a new conflict with a third aircraft.
The secondary-conflict-aware planner should reject the unsafe maneuver.

### `weather_replanning.json`

A weather zone becomes active on the nominal route.
The system should generate and apply a reroute around the weather zone.

### `no_conflict.json`

Aircraft remain safely separated.
The system should not generate false positive conflicts.

## Agents

The project includes real Jason / AgentSpeak(L) agent files under:

```text
src/main/agents/
```

Agents include:

* `aircraft.asl`
* `sector_controller.asl`
* `conflict_detector.asl`
* `resolution_planner.asl`
* `explanation_agent.asl`

The agents model BDI concepts:

* beliefs;
* goals;
* intentions;
* plans;
* message passing;
* delegation.

Jason source validation is performed through a lightweight smoke analyzer.
This ensures that CI can verify the existence and structure of the AgentSpeak files without requiring a fragile full runtime integration.

Run the smoke check:

```bash
./gradlew runJasonSmoke
```

## Suggested Demo Flow

1. Run `simple_conflict.json`.
2. Show the detected conflict.
3. Show the generated plan.
4. Show the applied maneuver in the GUI.
5. Run `emergency_priority.json`.
6. Explain how priority affects the selected maneuver.
7. Run `secondary_conflict.json`.
8. Show that unsafe secondary-conflict maneuvers are rejected.
9. Run `weather_replanning.json`.
10. Show weather-zone activation, reroute, waypoint, and trail deviation.
11. Open the explanation panel in the GUI.
12. Show raw JSONL events to demonstrate observability.
